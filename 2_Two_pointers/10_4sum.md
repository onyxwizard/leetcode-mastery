### 📘 Chapter: Two Pointers  
### 📌 Problem 10: 4Sum (LeetCode 18)

---

**Input**  
- `nums`: an integer array of length `n`  
- `target`: an integer

**Output**  
- A list of **all unique quadruplets** `[nums[a], nums[b], nums[c], nums[d]]` such that:
  - `0 <= a, b, c, d < n` and they are all distinct.
  - `nums[a] + nums[b] + nums[c] + nums[d] == target`.  
- The quadruplets can be returned in any order, but must not contain duplicates.

**Constraints**  
- `1 <= nums.length <= 200`  
- `-10⁹ <= nums[i] <= 10⁹`  
- `-10⁹ <= target <= 10⁹`

**Follow-up**  
- The natural follow‑up is to generalise to `kSum`; the optimal solution for 4Sum is **O(n³)** (or **O(n² log n)** with hashset), but the most interview‑friendly and robust is O(n³) two‑pointers.

---

### 🧠 Why this Approach?

4Sum is a direct extension of 3Sum and Two Sum II.  
- **Sorting** the array allows us to use the **two‑pointer technique** for the innermost two numbers, reducing a potential O(n⁴) brute force to **O(n³)**.  
- Handling duplicates is identical to 3Sum: after sorting, duplicate values are adjacent, so we can skip them in outer loops and while advancing pointers.  
- Using a **HashSet** to store pair sums (O(n²)) can theoretically solve 4Sum in O(n²) average time, but deduplication of quadruplets becomes extremely tricky without also sorting, and the two‑pointer method is simpler and guaranteed O(n³), which is perfectly acceptable for `n ≤ 200`.  

Thus, the **nested loops + two‑pointers** approach is the cleanest and most predictable solution for an interview.

---

### 🔨 Brute Force Approach (Four Nested Loops)

**Method:**  
Check all quadruplets `(i, j, k, l)` with `i < j < k < l`. If sum equals target, add the sorted quadruplet to a `HashSet` to ensure uniqueness, then convert to list.

**Time:** O(n⁴)  
**Space:** O(n) for the set of quadruplets.

```java
public List<List<Integer>> fourSum(int[] nums, int target) {
    Set<List<Integer>> set = new HashSet<>();
    int n = nums.length;
    for (int i = 0; i < n; i++) {
        for (int j = i + 1; j < n; j++) {
            for (int k = j + 1; k < n; k++) {
                for (int l = k + 1; l < n; l++) {
                    if ((long)nums[i] + nums[j] + nums[k] + nums[l] == target) {
                        List<Integer> quad = Arrays.asList(nums[i], nums[j], nums[k], nums[l]);
                        Collections.sort(quad);
                        set.add(quad);
                    }
                }
            }
        }
    }
    return new ArrayList<>(set);
}
```

Too slow for `n = 200` (n⁴ = 1.6×10⁹ operations), but mentioned as baseline.

### ⚡ Optimized Approach 1 – Sorting + Two Pointers (O(n³) time)

**Method:**  
1. Sort `nums`.  
2. Two outer loops for the first two numbers `i` and `j` (`i < j`).  
3. For each pair, set `left = j + 1`, `right = n - 1`, and find pairs that sum to `target - nums[i] - nums[j]` using two pointers, just like 3Sum.  
4. Skip duplicates for `i`, `j`, `left`, `right` to avoid duplicate quadruplets.  
5. Use `long` for sum comparisons to avoid integer overflow (since values up to ±10⁹, sum of four can exceed 2³¹-1).

**Time:** O(n³) – two outer loops O(n²) and inner two‑pointer O(n).  
**Space:** O(1) extra (ignoring output list). Sorting may use O(log n) stack.

```java
import java.util.*;

public List<List<Integer>> fourSum(int[] nums, int target) {
    List<List<Integer>> result = new ArrayList<>();
    if (nums == null || nums.length < 4) return result;
    Arrays.sort(nums);
    int n = nums.length;

    for (int i = 0; i < n - 3; i++) {
        if (i > 0 && nums[i] == nums[i - 1]) continue; // skip duplicate i
        for (int j = i + 1; j < n - 2; j++) {
            if (j > i + 1 && nums[j] == nums[j - 1]) continue; // skip duplicate j
            int left = j + 1, right = n - 1;
            long target2 = (long) target - nums[i] - nums[j]; // avoid overflow
            while (left < right) {
                int sum = nums[left] + nums[right];
                if (sum == target2) {
                    result.add(Arrays.asList(nums[i], nums[j], nums[left], nums[right]));
                    // skip duplicates for left and right
                    while (left < right && nums[left] == nums[left + 1]) left++;
                    while (left < right && nums[right] == nums[right - 1]) right--;
                    left++;
                    right--;
                } else if (sum < target2) {
                    left++;
                } else {
                    right--;
                }
            }
        }
    }
    return result;
}
```

### ⚡ Optimized Approach 2 – HashSet for Pair Sums (O(n²) average)

**Method:**  
- Use a `HashSet` to store all possible two‑sum complements as we iterate.  
- For each pair `(i, j)`, we look for the complement `target - nums[i] - nums[j]` among previously seen pairs, but careful not to reuse indices.  
- This requires a way to store pair sums along with their indices (e.g., `Map<Integer, List<int[]>>`) and then combine them, ensuring distinct indices and uniqueness of quadruplets.  
- While it can achieve O(n²) time, the code is significantly more complex and still requires sorting or a set for quadruplet deduplication.

Due to increased complexity and potential for bugs, it is rarely the first choice for 4Sum when `n ≤ 200` and O(n³) is acceptable. I'll mention it but not provide the full implementation.

### 📊 Solution Comparison & Trade‑offs

| Solution                     | Time    | Space   | Deduplication            | Notes |
|------------------------------|---------|---------|--------------------------|-------|
| Brute force (4 nested loops) | O(n⁴)   | O(n)    | Using Set                | Impractical. |
| Sorting + Two Pointers       | O(n³)   | O(1)    | Skip adjacent duplicates | **Gold standard** – simple, predictable. |
| HashSet of pair sums         | O(n²) average | O(n²)   | Complex, extra storage   | Overkill for n=200, tricky to get right. |

**Trade‑off:**  
- The sorting+two‑pointer approach is consistently chosen because it is easy to reason about, follows naturally from 3Sum, and the O(n³) runtime is well within limits (200³ = 8 million operations).  
- The O(n²) hash map approach is faster theoretically but uses O(n²) extra space and requires meticulous handling to avoid index reuse and duplicate quadruplets. In an interview, it’s best to present the two‑pointer method as the primary solution, and only mention the hash map alternative to show broader knowledge.

### 🎯 What to Present to the Interviewer

1. Recognise that 4Sum is a direct extension of 3Sum.  
2. Propose **sorting** to enable the two‑pointer technique for the inner two numbers.  
3. Walk through the algorithm: two fixed loops for `i` and `j`, then two‑pointer for `left` and `right` searching for the remaining sum.  
4. Emphasise **overflow handling** by casting to `long` when computing the target for the two pointers.  
5. Explain the **duplicate skipping** logic for `i`, `j`, `left`, and `right`.  
6. State complexities: O(n³) time, O(1) extra space.  
7. Optionally mention that a more advanced O(n²) approach using a hash map exists but is overcomplicated for the given constraints.  
8. If asked, generalise to `kSum` by using recursion with a base case of `2Sum` (two‑pointer) and pruning.

**One‑sentence summary:**  
*Reduce 4Sum to 3Sum by fixing the first element, then use sorting and two pointers to find the remaining three numbers, managing duplicates and overflow to get all unique quadruplets in O(n³) time.*
